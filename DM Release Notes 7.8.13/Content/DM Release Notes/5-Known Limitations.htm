5. Known Limitations

Following are the limitations identified during the current release.

Description

Required User Actions

While resubmitting a Model Mapping and Revision Tasks, some data (rejected by, rejection reason) under Notifications are missing in the previous rejected notification (if this notification has not been dismissed yet).

 

In case of exporting test cases before repair, where the conclusion fact type name is changed in the exported filename, the latest conclusion fact type name is displayed even though the test group is not repaired. In case of exporting again after repair, the latest exported file will overwrite the previous file.

As a workaround, the user can

export before repair>download>rename the file.

Then when exporting after repair, the file will not be overwritten.

Exception can occur when a user edits a value in the testing panel immediately after copying or cloning the asset which has a default Test Group, and it is not persistent (in the source asset).

After refreshing the page, the user can continue working as usual.

When the system is configured to use Rule ID configuration, the rule family row action to generate a new Row ID is currently not supported.

To generate a new Rule ID for an existing rule a new row should be created that matched the row logic and the prior one deleted.

On the export history compare, no results are displayed if executing a compare on the first deployment to an environment. The compare will only show the deployment differences when a prior deployment to the same environment exists.

 

The 'System Admin E-mail Address' under System Options is not available in DM.

 

You can view the “Something Went Wrong” message in some areas of the system as a generic error detection. These will be addressed with specific error handling in future releases. There is no risk of data integrity, but to resume the application, it needs to be reloaded to be in a consistent state to move forward.

If it keeps occurring, you can log out and login manually to return to home. If it still occurs and prevents you from moving forward, open a defect.

The Input field under Test Case filter, to search Test Cases by Fact Type values is not available for SQL Server database due to limited JSON search capabilities.

Note: SQL database is not supported from DM 7.7 release.

 

Test Group import files are limited to 1 megabyte in size. Attempting to import a Test Group size larger than 1 megabyte may result in an exception.

The default Test Group size can be overridden by adding the following properties in dt.application.properties:

  • spring.servlet.multipart.max-file-size=<max file size>

  • spring.servlet.multipart.max-request-size=<max file size>

For example,

  • spring.servlet.multipart.max-file-size=10MB

  • spring.servlet.multipart.max-request-size=10MB

Large Test Group imports may timeout even before the progress bar is displayed in the user interface.

Note: The import will still run in the background.

The default timeout can be increased by adding the following properties in dt.application.properties:

  • dt.http.timeout.default=<time in milliseconds>

  • spring.mvc.async.request-timeout=<time in milliseconds>

For example,

  • dt.http.timeout.default=60000

  • spring.mvc.async.request-timeout=60000

The indicator to notify the users if they are not the assignee of the Fact Type tasks will be implemented in future as it is different than the Modeling task.

 

The Function display is not updated immediately, it gets updated only when you navigate to some other screen and navigate back to the Functions page.

 

When editing an asset to a task or cloning an asset that has BIM entities and Test Cases that require repair, the test case BIM entity association will not be updated to the new task asset BIM entities.  This reassociation can only occur for Test Cases that do not require repair.  If the new task asset has a BIM entity renamed, the test case repair operation will not be able to find the associated BIM entities and the BIM related data in the test case will be lost.

Ensure all the Test Cases that are expected to be used with new versions or clones of an asset are repaired so that they can be properly copied to a task with an edited or cloned asset.

Decision Compare currently does not support the compare of Knowledge Model differences.

 

If certain test cases contain a multi-level BIM structure where some Fact Types inside a BIM instance have populated values, but other instances do not (i.e., not populated or null), applying a filter on that Fact Type with the Not Populated checkbox selected may not return the expected results. This is a null check limitation across the BIM structure and applies only to Oracle databases.
