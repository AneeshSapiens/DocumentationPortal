7.1 Current Upgrade Considerations

The following upgrade considerations have been implemented in the 7.8.15 release:

  1. Optimize DT scaling on Kubernetes

    This release introduces key improvements in memory and CPU utilization for DT applications and provides enhanced control through new configuration properties.

    The following properties have been added to the dt-repo, dt-generate, dt-exec and dt-repair application configuration maps:

    • tg.progress.update.seconds

    • tg.execution.result.buffer

    • tg.execution.result.concurrency

    • tg.generate.result.buffer

    • tg.generate.result.concurrency

    • tg.repair.result.buffer

    • tg.repair.result.concurrency

    Note: The following properties were available in previous releases and must be removed from the dt-repo, dt-generate, and dt-repair application configuration maps:

    • tg.repair.concurrency

    • tg.generate.concurrency

    • dt.repo.save.tc.generated.concurrency

    • dt.repo.save.tc.exec.concurrency

    Refer to the Decision Testing (DT) Advanced Configuration section in Decision Manager (DM) Configuration Guide for more details.

  2. Change in Flow Association Behavior

    Until DM V7.8.12, Flows could be associated with multiple communities at the same time. This allowed a single Flow to exist in more than one community.

    As of DM V7.8.15, Flows are now restricted to a single community. While existing associations will remain unchanged, any new Flow associations must comply with this restriction.

    Users are advised to review their existing Flows and ensure that each Flow is associated with only one community. While this is not mandatory, addressing such cases can help prevent validation errors when modifying Flow associations in the future.

    To identify Flows currently linked to multiple communities, users can run the following SQL query:

    Copy 
    SELECT DISTINCT rfd.business_name AS flow_Name,
        vg.NAME AS community_Name
    FROM RULE_FLOW_VIEW_GROUP rfvg
    INNER JOIN RULE_FLOW rf ON rf.ID = rfvg.RULE_FLOW_ID
    INNER JOIN ASSET a ON a.ID = rfvg.RULE_FLOW_ID
    INNER JOIN RULE_FLOW_DEFINITION rfd ON rf.RULE_FLOW_DEF_ID = rfd.ID
    INNER JOIN VIEW_GROUP vg ON vg.ID = rfvg.VIEW_GROUP_ID
    WHERE rf.RULE_FLOW_DEF_ID IN (
        SELECT RULE_FLOW_DEF_ID
        FROM RULE_FLOW
        WHERE ID IN (
            SELECT RULE_FLOW_ID
            FROM RULE_FLOW_VIEW_GROUP
            GROUP BY RULE_FLOW_ID
            HAVING COUNT(VIEW_GROUP_ID) > 1
        )
    )
    AND a.state IN (2,4)
    ORDER BY rfd.business_name, vg.NAME;

     

    The output of the query will be in the following format if there are Flows existing in multiple communities.

    flow_name

    community_name
    Policy Renewal Method Flow P&C Underwriting Community
    Policy Renewal Method Flow XYZ Bank
    Policy Auto Insurance Underwriting Flow P&C Underwriting Accelerator Community
    Policy Auto Insurance Underwriting Flow Sample Community

     

    • If the query returns results, users can create dissociation requests in the Repository Task to remove the Flow from the communities where it is not needed.

    • If no results are returned, no action is needed.

    While this step is optional, proactively resolving multiple-community associations will help avoid validation errors when making changes to Flow associations in the future.
