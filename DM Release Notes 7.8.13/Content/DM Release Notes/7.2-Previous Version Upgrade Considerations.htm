7.2 Previous Version Upgrade Considerations

Version Upgrade Considerations for DM 7.8.4 release

SSO Updates

An SSO enhancement was implemented to control encryption certificate injection based on the following property in DM:

  • decision.saml2.sp.authn.requests.signed

If the value of the above-mentioned property is true, the service provider expects the response to be encrypted with a certificate included in the metadata.

And if the value is false, no encryption is expected, and metadata will not include the encryption certificate.

Added the following property to inject value:

  • decision.base.url=<Scheme>://<DomainName>

where,

  • Scheme = http or https

  • DomainName = dnsname or host:port

Note: The following properties were available in previous releases and have now been replaced with the above-mentioned property (decision.base.url) and must be removed:

  • bdms.cluster.frontend.scheme

  • bdms.cluster.frontend.host

  • bdms.cluster.frontend.port

Note: For the above changes to take effect, refresh the metadata from DM in the IDP.

 

Version Upgrade Considerations for DM 7.7.7 release

SSO Configuration Updates

With this upgrade, you can include absolute URL and signing certificate in metadata. You can send signed requests to IDP when DM is the service provider.

Add the following properties and set the value to ‘true’ to enable SSO with certificate information:

  • decision.saml2.idp.authn.requests.signed=false

  • decision.saml2.sp.authn.requests.signed=false

Note: The default value is false.

The following are the mandatory parameters to get the service provider metadata with authentication request signing option turned on under bdms.cust.properties:

  • decision.dm.saml2.sp.keystore.filename=<keystore file name>

  • decision.dm.saml2.sp.keystore.alias=<keystore alias name>

Also add the following keystore password in {decision_home}/secureconf/secureargs.properties file:

  • decision.dm.saml2.sp.keystore.password=<keystore password>

Refer to the Customize DM SSO configurations section in Decision Manager (DM) Configuration Guide to configure signed SAML authentication request.

Prior to DM V7.7, all UDFs had to be compiled to be compatible with Java 8. As of DM V7.7, UDFs must be compiled to be compatible with Java 17 for use with DM and DE. If the Java Compilation Version in your Java Format Adapter settings is set to 8, you will need a version of the UDF jar that is compatible with Java 8 for use with your Java Format Adapter deployed code. Either compile with a Java 8 JDK or use javac --release 8.

Version Upgrade Considerations for DM 7.7.3 release

Complete the following when you are upgrading from 7.6 to 7.7:

  1. The file canonicalization cache was added back in JDK 1.4.2 to improve startup time. In JDK 12 the file canonicalization cache by default disabled, this in turn creates a performance degradation in the server startup. However, the file canonicalization cache can still be enabled by setting the "sun.io.useCanonCaches" system property. This is merely a change to the default value.

    To ensure system startup performance, follow the installation information given below to enable the file canonicalization cache:

  • For bundle installation, if it does not already exist, add the following to decision_home}/bin/common.sh:

    JAVA_OPTS="$JAVA_OPTS -Dsun.io.useCanonCaches=true"

  • For war installation, add it as any other JVM property.

  1. The code generation logic relies on the reflection API features of older JDK versions, which are now not supported by JDK-17 out of the box, however they can be enabled by setting the following Java tool options. For bundle installation, if these do not already exist, add the following to {decision_home}/bin/common.sh:

  • JAVA_TOOL_OPTIONS=--add-opens=java.base/sun.reflect.annotation=ALL-UNNAMED

  • export JAVA_TOOL_OPTIONS

  • JAVA_OPTS="$JAVA_OPTS -XX:+UseParallelGC"

  • JAVA_OPTS="$JAVA_OPTS --add-modules=java.se"

  • JAVA_OPTS="$JAVA_OPTS --add-exports=java.base/jdk.internal.ref=ALL-UNNAMED"

  • JAVA_OPTS="$JAVA_OPTS --add-opens=java.base/sun.nio.ch=ALL-UNNAMED"

  • JAVA_OPTS="$JAVA_OPTS --add-opens=java.base/java.lang=ALL-UNNAMED"

  • JAVA_OPTS="$JAVA_OPTS --add-opens=java.management/sun.management=ALL-UNNAMED"

  • JAVA_OPTS="$JAVA_OPTS --add-opens=jdk.management/com.sun.management.internal=ALL-UNNAMED"

    For more information on the parameters, refer to the setenv.sh section under Configuration Files in the Decision Manager (DM) Installation Guide.

  • For war installation, add these as any other JVM property.

Java Upgrade

  • Decision Manager has been upgraded to Java 17. The deployment environment must now run with a Java 17 JVM.

  • The Java deployment adaptor can deploy compiled Java code as Java 8 or Java 17. If Java 17 is the preferred Java adaptor deployment, update your existing environments to set the Java compilation version to 17. Java 6 and 7 are no longer supported.

  • DM and DT have been upgraded to work with JDK 17 onwards which requires at least Tomcat version 10 as an application server container.

MS-SQL

  • MS-SQL database is not supported for DM version 7.7.3. The support for the MS-SQL database will be provided in a future release.

Oracle DB Connection by URL

Due to changes from a custom in-house JDBC-r2dbc-based driver for Oracle to an Oracle-provided r2dbc driver, this enhancement was done for support-based connection to Oracle. For this support-based connection, we need the following properties:

  • If you are configuring repo.jdbc.url, then the decision.repo.flyway.jdbc.url is a mandatory parameter to run DB migrations successfully. Where: decision.repo.flyway.jdbc.url=jdbc:oracle:thin:@<hostName>:<port>/<ServiceName>

Incorrect Display Name Change for conflicting UDF and new Core Sapiens Functions

If any of the three UDFs DATE, DATETIME, or LAST_DAY_OF_MONTH are defined in your instance of Sapiens Decision, the Display Names for these UDFs are not updated as expected with the addition of these functions as new core functions. Only the Display Name of the UDF is updated with _UDF, while the Name field of the function is not updated with _UDF. It is important to correct the Display names as follows:

  • Rename DATE_UDF to DATE

  • Rename DATETIME_UDF to DATETIME

  • Rename LAST_DAY_OF_MONTH_UDF to LAST_DAY_OF_MONTH

 

Validating the Rule Family using the above functions is giving the following validation errors if the names are not corrected. For Example:

 

  • Function "DATE_UDF" does not exist.

  • Function "DATE_UDF" has Invalid number of parameters expected "0" but found "3".

 

Update the Display Name of the UDF manually to remove “_UDF” to validate and execute the UDF successfully.

 

It may be necessary to update the UDF name to another name first and then back to its correct name to adjust the UDF Display Name appropriately.

User and System Functions

If the DM version 7.3.7.2 or above is already consumed and any of the below functions has already been renamed to _UDF (in case of conflicts between user and system functions), then for those instances the migration will not do anything. The user is expected to fix it manually from UI with the required permissions.

  • IS_DATE

  • IS_REGULAR_EXPRESSION_MATCH

  • MIN_DATE

  • MAX_DATE

  • DATETIME

  • DATE

  • DATE_FROM_DATETIME

  • LAST_DAY_OF_MONTH

 

The following logic change was introduced as part of 7.3.10 release for Decision provided function Name (system function):

  • If you are in between DM versions 7.3.7.2 and 7.3.9.1, and there are user functions with the same name as the newly introduced system functions, then the user function will be renamed as *_UDF and system function will remain as is.

  • If you are upgrading to DM version 7.3.10, and there are user functions available with the same name as the newly introduced system functions, then the user function’s name will not change but the system functions will be added as *_CORE.

 

In order to apply the above logic change, the existing migration was updated (introduced as part of DM versions between 7.3.7.2 to 7.3.9.1). To rerun the migration successfully, the existing migration entries must be deleted from the database.

 

Note: The above logic change impacts the users of DM version 7.3.7.2 and above.

 

Perform the following steps manually before upgrading:

  • To identify the number of migration entries to be deleted, run the below query.

 

  • select count(*) from "ALL_SCHEMA_VERSION" where "version" >= '2007030700004';

  • To delete the migration entries (in case the above query returns >0):

 

  • delete from "ALL_SCHEMA_VERSION" where "version" >= '2007030700004';

  • After deleting, the above SELECT query should return 0.

Version Upgrade Considerations for DM 7.3.9.1 release

The system function INDEX is a part of the 7.3.9.1 release. If a user function is already available prior to migrating to 7.3.9.1 with the same name as INDEX, then the migration will insert the system function as INDEX_CORE and the user function will remain untouched. Due to certain technical limitations, the IS_UDF Identifier in the database and UI is getting set to TRUE for system defined INDEX (or INDEX_CORE) function and this function will be considered as a user function.

To fix this system function issue, complete the following steps:

Before the upgrade:

Run the below SELECT query

  • select count(*) from function_md fm where fm.name = 'INDEX';

After the upgrade:

Based on the result of the above, complete the following steps after the upgrade:

 

If the above query returns no row

If the above query returns 1 or more rows

For SQL and ORACLE DB

  1. Run the below UPDATE query:

  • UPDATE function_md set is_udf = 0 where name = 'INDEX';

 

  1. Validate using below query:

  • SELECT count(*) from function_md fm where fm.name = 'INDEX' and fm.is_udf = 0;

The query should return count as 1.

 

For PostgreSQL

  1. Run the below UPDATE query:

  • UPDATE function_md set is_udf = false where name = 'INDEX';

 

  1. Validate using below query:

  • SELECT count(*) from function_md fm where fm.name = 'INDEX' and fm.is_udf = false;

 

The query should return count as 1.

For SQL and ORACLE DB

  1. Run the below UPDATE query:

  • UPDATE function_md set is_udf = 0 where name = 'INDEX_CORE';

  1. Validate using below query:

  • SELECT count(*) from function_md fm where fm.name = 'INDEX_CORE' and fm.is_udf = 0;

The query should return count as 1.

 

For PostgreSQL

  1. Run the below UPDATE query:

  • UPDATE function_md set is_udf = false where name = 'INDEX_CORE';

 

  1. Validate using below query:

  • SELECT count(*) from function_md fm where fm.name = 'INDEX_CORE' and fm.is_udf = false;

 

The query should return count as 1.

Dropping of legacy testing tables

Upgrade Type: Migration

Description: With V7.3, the legacy testing tables from V6 that are now replaced with the new testing module will be dropped. By default, on start-up of V7.3, the application will query the legacy testing tables to confirm that all test groups have been migrated successfully. If any test group has not yet been migrated or had a failed migration, the legacy test tables will not be dropped, and the application will not start.

There are options for a client to override the default behavior and select whether to drop the tables if there are any unmigrated or failed migrations or retain the legacy test tables.

The DROP option should be selected if a client feels they do not need to complete the outstanding migrations or address any failed test group migrations. Note that some outstanding migrations may be Rule Family branch test groups that have not been migrated as Rule Family branch test groups are not currently a feature in V7.

The RETAIN option should be selected if a client has a significant number of Rule Family branch test groups they feel they need to retain for a future migration of this data.

To assist in evaluation of the desired option, before upgrading to V7.3, the client may run the following queries in their existing database:

  • Identify number of Rule Family branch test groups.

    Select count(*)

    from testing_groups

    where testing_group_type =2

  • Identify non-migrated test groups (excluding Rule Family branch test groups)

    select case

    when migrated_to_dt = 0 then ‘Not migrated’

    when migrated_to_dt = 2 then ‘Failed’

    end as migrated_to_dt,

    count(*)

    from testing_groups

    where testing_group_type !=2

          and (rule_family_view_id is not NULL or decision_view_id  is not NULL or decision_flow_id is not NULL)

          and migrated_to_dt in (0,2)

    group by migrated_to_dt

    order by migrated_to_dt;

Based on the above queries, a decision can be made to drop all the legacy test tables which will greatly improve the database sizing or retain the legacy tables.

If there are no non-migrated test groups (including Rule Family branch test groups) or failed test groups, start up the system with default behavior so that all the legacy test tables will be dropped.

If there are non-migrated test groups, ensure you have run the migration web services mentioned in the prior Upgrade Consideration section (Manual Migration of Test Groups Required) under V7.0.2.1.

If there is not a significant number of non-migrated (including Rule Family branch test groups) or failed test groups that the business feels a necessity to retain, set the following bdms.cust.property in the bdms.cust.properties file before starting V7.3:

  • bdms.drop.test.tables.override.unmigrated.testgroups=DROP

If there are a significant number of Rule family branch test cases, that are worth retaining for future migration, set the following bdms.cust.property in the bdms.cust.properties file before starting V7.3:

  • bdms.drop.test.tables.override.unmigrated.testgroups=RETAIN

Deployment code generation change

Code generation defect correction

There was a change to the Java and DE code generation in this release to address an issue with the Java and DE trace values for Flows.

 

In some situations, with non-root BIM entities and where multiple Decisions exist in the Flow with the same conclusion but different views, the first Decisions executed may show results of the final Decision rather than the specific Fact Type values at the point of execution.

 

There are no issues with the final Flow Fact Type values, however if the values in the trace at the time of a Decision execution within the Flow are important data of interest, it will be necessary to regenerate those Flows to ensure that the Flow trace data is showing the actual Fact Type values for each Decision at the point it was executed within the Flow rather than the final Fact Type values.

Refer to the Fix updated for DECISION- 28590 for additional details.

 

If a client uses Flows with BIM entities that also have multiple instances of the same Decision conclusion with different views in the same Flow and are interested in capturing interim Decision trace details at the point of Decision execution within the Flow, it will be necessary to regenerate and redeploy these types of Flows.

Migration

Manual Migration of Test Groups Required

As part of V7.0, the Test Facility was re-architected for performance scalability. As a result, Test Groups will need to be migrated from prior version data structure to the new Test Facility data structure. Given these migrations can take considerable time depending on the number of test groups existing in each client environment, the Test Group migrations are not part of the automated upgrade migrations that occur on startup of the V7.0 server for the first time, so that clients will not need to wait for an extended period for V7 to start up and be ready for use.

The Test Group migrations are run with the provided web service that will allow clients to run the migrations in chunks with set time durations until complete. This will allow clients with a large number of Test Groups to run the migration over a number of evenings or weekends until complete.

The web service can be run by a user with ADMIN permission from the Administration -> Utilities -> Web Services Management page.

To run the Test Group Migration, enter the following web service in the Web Service Input field:

ws/api/testing/migrateToDt/start?
timeWindowMinutes=120

 

The minutes parameter above is set to run the job for up to 120 minutes. This parameter can be set to the duration for which the job should run. If running overnight or on a weekend this value can be set to multiple hours to complete all the migrations as soon as possible. This web service can be run multiple times until all Test Groups are migrated and will continue from the Test Groups not yet migrated.

 

If the timeWindowMinutes parameter is omitted the web service will run for 120 minutes and then stop.

 

Note: The environment endpoint is set automatically for the application in the Web Service URL field on the page.

 

At any time, the migration job can be stopped by running the following web service:

 

ws/api/testing/migrateToDt/stop

 

To check the progress of the migrations, run the following web service:

 

ws/api/testing/migrateToDt/progress

 

All Test Cases should migrate, but in the case of an exception, failed test groups are marked in the database as failed. Failed Test Cases will be skipped on subsequent submissions of the Test Group migration web service. The following parameter can be added to the web service to rerun failed Test Cases:

 

includeFailed=true

 

Example:

ws/api/testing/migrateToDt/start?
timeWindowMinutes=120&
includeFailed=true

 

  • Must run the migration web service to migrate prior version Test Groups. Assets will no longer have the test groups from the prior version until the migration is executed.

Java Date Format

The default approach to the Java Date for V7 has changed

The Decision default for versions prior to V7 was to utilize a java.util.Long (e.g. timestamp) date format for Decision execution in the Decision tester, generated Java or Decision Execution (DE). As of V7, the Decision default is to use java.util.Date for date formats in Decision execution.

 

The bdes.export.data-type.temporal property is used to designate the approach to handling temporal data types (‘DATE’, ‘DATE_TIME’) defined for Fact Types in Sapiens Decision at execution time. This will control the Java data type used in interfaces to User Defined Functions (UDFs) and deployed code interfaces provided by the Deployment Adapters for the temporal data types.

 

If DATE and DATE_TIME are to be implemented as ‘java.util.Long’ (e.g. timestamp) the property should be set in the bdms.cust.properties file as:
bdes.export.data-type.temporal=timestamp

 

If DATE and DATE_TIME are to be implemented as ‘java.util.Date’ the property shall be set in the bdms.cust.properties file as:
bdes.export.data-type.temporal=date

 

Note: If no value is set for the property, the implementation will now default to java.util.Date as if the property was set as:

bdes.export.data-type.temporal=date

 

The following property files may need to be updated to utilize the correct date functions.

 

If
bdes.export.data-type.temporal=timestamp, then bdes.function.map.cust.properties and java.function.map.cust.properties can be empty or have the following entries:

  • NOW=NOW

  • TODAY=TODAY

 

If
bdes.export.data-type.temporal=date, then the following entries in bdes.function.map.cust.properties and java.function.map.cust.properties can be empty or have the following entries:

  • NOW=NOW_DATE

  • TODAY=TODAY_DATE

Note: It is best practice to leave the above files empty rather than including the above entries.

  • No change required unless you use java.util.Long date format.

  • If so, you can change the property to continue with java.util.Long or change UDFs to use java.util.Date.

 

 

 

 

Default DM URL

The Default URL for Decision is changed

The default URL for the application endpoint will now have a /dm as part of the URL. This is done to improve security and support application transaction filtering.

 

For example, the new URL will be:

 

https://dm.sapiens.com/dm/login.jsp

 

This is a changed from prior versions:

 

https://dm.sapiens.com/login.jsp

 

If you do not use the included Tomcat bundle provided with the installation package, you will also need to make some configuration changes in Tomcat configuration files. Refer to the following documents for installation and configuration instructions:

 

  • Decision Manger (DM) Installation Guide 7.0

  • Decision Manager (DM) Configuration Guide 7.0

 

The URL can be reset to not have the /dm and continue with the same URL as prior versions if desired. To configure Decision to have the same URL as prior versions refer to the following document section:

 

Decision Manager (DM) Configuration Guide 7.0

Section 11 – Configure Custom Context Path

 

  • If the provided Tomcat Bundle is used for installation no change is required, other than notification of the new URL.

  • If the provided Tomcat bundle is not used for installation modifications to Tomcat configurations are required.

  • If there is a need to retain the original URL, this can be done via configuration.

Deployment

Deployment with CSV format Test Groups is not supported

With V7, CSV Test Group export is not supported. CSV Test Group and Test Case export will be supported in a future release.

 

If an environment is configured to Include Test Groups and the Test Group Format is set to CSV, the Include Test Group option will be automatically disabled in the V7 upgrade migration process.

 

This will ensure that deployments will not fail. However, since CSV Test Group export is not supported in V7, CSV Test Groups are not supported in the deployment package.

 

The environment can be updated to export Test Groups in JSON or XML format to include Test Groups in the deployment package.

  • Deployments will not support the inclusion of Test Groups in the deployment in the CSV format.

Deep Links

Deep Link Changes

In versions prior to Version 7, deep links could be generated from the UI for a Decision Flow, Decision, or a Rule Family to provide a URL for DM that could be used to directly access the asset without navigating through the DM UI.

 

With Version 7, the format of the Deep Link URLs has changed, and URLs generated in prior versions will no longer work with Version 7.

 

Version 7 has also extended the Deep Links capability to support copying the URL for any page in the application from the browser address field to directly access that page.

 

All Ids referenced within prior Deep Links will remain the same in Version 7 so Deep Links generated using the “Direct Links via IDs” feature in prior versions may be converted to the new format either manually or programmatically.

 

For example, consider the “Direct link via IDs” prior and new formats for the following assets:

 

Decision Flow Prior Deep Link:

{DM_URL}/shell.html?type=RULE_FLOW&oper=distinct&id=1021532&
viewgroupid=1014298

 

Decision Flow New Deep Link:

{DM_URL}/dm/enterprise/index.html#/viewGroups/1014298/decisionFlow/1021532

 

Decision Prior Deep Link: {DM_URL}/shell.html?type=DECISION_VIEW&oper=distinct&id=1018646&
viewgroupid=1014298

 

Decision New Deep Link:

{DM_URL}/dm/enterprise/index.html#/
viewGroups/1014298/decision/1018646

 

Rule Family Prior Deep Link:

{DM_URL}/shell.html?type=
RULE_FAMILY_VIEW&oper=distinct
&id=1018672&viewgroupid=1014298&
arented=1018646

 

Rule Family New Deep Link:
{DM_URL}/dm/enterprise/index.html#/
viewGroups/1014298/decision/
1018646/table/1018672

 

  • If clients used and persisted Deep Links to DM, they must obtain new Deep Links for the assets or convert prior Deep Links to the new Deep Link format.

Deployment

Include Decision Verb in Deployment File Names

With Version 7 support for graphical compatibility with the Decision Modeling and Notation (DMN) the Decision Verb is no longer part of a Decision name.

 

A Decision Verb of “Determine” is created with Decisions to support compatibility with export adapter files where the Verb was supported in the past to ensure backward compatibility.

 

For some adapters such as Minimal CSV, Modular CSV, Modular XLS, and Modular XML the Decision Verb was also used in file names. The default in Version 7 is to not include the Decision Verb in file names, but if it is required, a new Format Adapter property “Exclude Verb from the Exported File” is included to allow clients that require the verb in the files names to set this to “No” to include the Verb in file names for prior version compatibility.

  • If the following Format Adapters: Minimal CSV, Modular CSV, Modular XLS, and Modular XML are used and file names with the Decision Verb are required, set the “Exclude Verb from Exported File” property to “No”.
