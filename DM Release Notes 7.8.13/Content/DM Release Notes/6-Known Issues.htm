6. Known Issues

Following are the issues identified during the current release.

Description

Required User Actions

Search/Filter Functionality is not working in some dialogs of the Administration pages.

No user action is required.

In the Flow change type identification, it shows version change even when it is an unchanged asset in the following scenarios:

1. Edit the Flow with a Decision in a task, and then edit Gateway Fact Type. Every asset is unchanged as expected.

2. Do a version change in a Fact Type and connect it to the Flow and Decision. All assets are recognized as version change as expected.

3. Connect the Decision and the Flow to the approved version (change the Fact Type version in a Gateway in the Flow and revert the Fact Type in Decision). Then validate the Decision and the Flow. Now the Decision is correctly recognized as unchanged, but the Flow is still marked as a version change when it is unchanged.

 

In the SQL Server database, the Decision Test Cases instance will be partially copied on edit or cloning of an asset if there is an error. The Test Cases that are copied will be shown in the new asset but those that were not copied will not be seen and the user won’t get any exception.

Note: SQL Server database is not supported from DM version 7.7.3. This known issue applies to prior versions and the next available versions that will support the SQL Server database.

 

In Oracle database, when editing or cloning an asset, the Decision Test Cases copy may fail with an exception when working with more than 25K Test Cases (for Flow - only Flow’s Test Cases are counted, for Decision – only Decision’s Test Cases are counted, and all its Rule Family’s Test cases are counted). The Test Cases that are copied will be shown in the new asset but those that were not copied will not be seen.

 

In the Additional Info screen, the DEC_READ, DF_READ, PROXY_INSTANCE_READ permissions are required to view Decisions/ Flows/ Knowledge Sources in the panel. It is different than Repository or Quick Search where the assets can be viewed with VG_READ permissions but cannot be opened without the DEC_READ, DF_READ, PROXY_INSTANCE_READ permissions.

 

While checking for Approvers, only the users with the corresponding Repository Task permission is displayed, but it is not verified if the Approver has access to the communities the assets are from or targeted to.

 

The results counter in Advanced Search, includes the assets from unauthorized communities and private rule families but the assets are not displayed in the results.

 

When reassigning a candidate Fact Type Task, the original submitter is included in the list of potential assignees even if they do not have the appropriate role for the current stage of the approval process

 

For all autocomplete fields, the width of the autocomplete box is determined by the content and is based on the longest string in the list. The string sometimes can also exceed the width of the dialog.

 

The 'not latest' indicators in the Rule Family List, Task (Rule Family and Decisions tab) may be different from what is displayed on the diagram.  The Diagram may need to be refreshed to get the latest indicators as the diagram caches some information for performance and usability.

 

While executing Test Cases, a deployment error is displayed instead of proper error when multiple Fact Types have the same normalized name. A normalized name is the Fact Type name without spaces or special characters not allowed in a Java property name removed.

Identify the Fact Types in the asset that have the same normalized name and change the name of one of them to ensure they are unique.

When you navigate to a Decision and open the Model.AI chat session and then create a new Decision from the Rule Family branch, the Decision Fact Type name is populated from the Model.AI chat, not the Rule Family you are attempting to branch.

Navigate back from the Decision diagram and open it again or refresh the browser and the Create New Decision From Rule Family Branch will display the correct drop-down values.

When you open the Testing panel of a Decision that was created using Model.AI, Fact Types which are not part of the Decision are displayed in the Testing panel.

Perform any changes/updates in the Decision and validate again to view the relevant Fact Types in the Testing panel of the Decision.

When you type any text in the description field of the Knowledge Model, the characters are removed.

 

Currently, if a Decision with a Knowledge Model is used in a Flow, the Decision must also be added to the release revision for the KM connection configuration to be deployed for the Decision.

 

If a Fact Type that is used in a Knowledge Model is added to the Decision Rule Family, and the Decision Rule Family is subsequently reverted, this may cause a Something went wrong exception.

The affected Decision may need to be removed from the task and added back to the task to continue.

Execution of the drop retained testing tables web service does not get logged with the default logging settings.

 

In the Postgres database, when filtering a test case in a test group, if a value is provided for Label, then removed, and the Not Populated option is selected in the Filter by Search Parameters dialog, the filter does not work as expected.

 

When filtered using an input of the Multiple Value List Indicator Fact Type and the Not Populated checkbox is selected, all test cases are displayed, even if they do not match the filter criteria.

 
The 'Recently Viewed' section on the Decision home screen displays Flow entries from all associated communities, even if accessed or viewed from only one community.  
When a Flow is associated with multiple communities, and one version of the Flow is retired, the retired asset should display all associated communities' information in the Advanced Search results. However, the Advanced Search currently displays information for only one community, not all associated communities.  
When a version of the Flow uses a Decision based on the Private to Community Rule Family, that version of the Flow is not associated.  
In the "Add Assets to Repository Task" dialog for the Association/Dissociation Assets Repository Task, an SWW (Something Went Wrong) error occurs when more than 10 flows are selected. This issue happens only when users update their selection by switching to the Source tab, return to the Selected Assets tab, and attempt to navigate through the previously unvisited pages (paginator). After navigating to the Selected Assets tab, if users want to change the selection before navigating to the Source tab, they must navigate through all pages (paginator) in the Selected Assets tab to avoid encountering the error.
In the "Add Assets to Repository Task" dialog for the Association/Dissociation Assets Repository Task, the Asset counter on the Selected Assets tab displays incorrect counts when more than 10 flows are selected. To ensure the Asset counts are accurate, users must navigate through all pages (paginator) in the Selected Assets tab.
In the "Add Assets to Repository Task" dialog for the Association/Dissociation Assets Repository Task, when more than 10 assets are selected, the Select/Unselect All Associate option works only once and does not update the selection for Assets on other pages. Users can manually select or unselect the checkbox for each Asset individually to complete the association.

In the Azure Kubernetes Service (AKS) environment, purging test cases and Knowledge Sources displays a "Purge in process" message with a success response in the UI. However, the asset count remains unchanged, indicating that the purge is not working as expected.

 

When a user adds a Flow for an Association/Dissociation task, the related Decisions are pulled into the task. If the user later removes the Flow and adds another Flow with common Decisions to the same task but for a different target folder, the common Decisions will not be pulled again. This happens because an existing Association request already includes those Decisions, preventing them from being reassociated with the new Flow. As a result, the user will be unable to proceed due to missing related Decisions for the newly added Flow.

To avoid this issue, users should first remove the associated Flow and then manually remove the related Decisions before adding a new Flow to the task. This ensures that all necessary Decisions are correctly pulled when adding the new Flow for the different target folder.
