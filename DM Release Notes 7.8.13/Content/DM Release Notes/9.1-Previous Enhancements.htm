Asynchronous Processing for Export Tag In Project and Enhanced Environment Search

The Export Tag In Project service, which was previously available only via SOAP and executed synchronously, has been enhanced to support both REST and SOAP with asynchronous processing. Instead of waiting for the export to complete, the API now returns an "Export Started Successfully" response and continues processing in the background.

Knowledge Model feature

A Knowledge Model is a modeling asset included in a Decision Model to represent any logic of a Decision that is not modeled in a Rule Family. It can be considered a black box to the Decision Model where Fact Types can be provided to the Knowledge Model as inputs and one or more conclusions can be returned from the Knowledge Model as outputs. Currently, any Knowledge Model can be implemented as any REST web service call to a machine learning algorithm or any web service that provides business logic.

Refer to Decision Manager (DM) User Guide > Knowledge Model

User Status Report – Permission Update Id column

The ‘Permission Update Id’ column is added in the “User Status Report” to indicate a user’s latest ‘Permission Update Id’.  The user ‘Permission Update Id” is used in conjunction with user and user roles SSO IDP auto synchronization.  A user’s profile in the IDP can be extended to include a ‘Permission Update Id’ and user community roles.  If a user’s roles are updated in the IDP, the user ‘Permission Update Id’ should also be updated and if it is different to the current value in Decision, the user IDP roles will be auto synched with the user’s latest authentication token data on login.  Typically, the “Permission Update id’ is set to a date or date time to indicate the change of user community roles to force a synchronization of updated user roles on next Decision Manager user login.

Refer to Decision Manager (DM) User Guide > Reports > Security > User Status Report

Improved RF Message Migration

Prior to Sapiens Decision Version 7, Rule Family message categories and messages belonged to the association of a Rule Family to a Decision. That is, the messages for a Rule Family were different based on the Decision with which it was associated. This provided additional flexibility for having different messages based on the usage of a Rule Family in different Decisions, but also meant that where messages should be the same across Decisions they would have to be manually synchronized. Based on analysis with clients and the fact that many clients treated a Rule Family message as a key part of the logic, in Version 7 Rule Family messages were moved to be part of the Rule Family. In situations where a message should be different for a specific Decision, the usage can be handled the same way as logic differences by introducing a new view (context) for the Rule Family. This now ensures a Rule Family view and version will have the same messages across Decisions and reduces maintenance overhead to ensure their consistency while still providing views to support differences when required.

For existing approved Decision versions their messages with associated Rule Families are preserved and will remain as they were at the time of their approval.

Given the difference in behavior with Version 7 there needed to be a point in which a Rule Family would have its messages migrated from Decisions to the Rule Family. For versions prior to Version 7.2.9 the following is the Decision to Rule Family message migration approach:

  1. New Rule Families will immediately own messages and they cannot be different across Decisions.

  2. For an approved Rule Family, its messages will be migrated to the Rule Family on the first change of the Rule Family.

  • Messages for the Rule Family will be based on the first Decision for which the Rule Family was edited.

Given the complexity of moving multiple different messages across Decisions to one set of messages for a Rule Family there were some limitations with the Rule Family message migration prior to 7.2.9.

  1. The first Decision for the Rule Family to be edited defines the Rule Family message categories and messages for the Rule Family. Message categories and messages on other Decisions would not be included with the new version of the Rule Family automatically. This required users to check the messages across all Decisions to decide the messages to retain and possibly the best Decision for which to first add to a task for the Rule Family messages. In addition, if one community did not have message categories these would be removed for the Rule Family.

  2. It was not clear to users when a Rule Family’s messages were being migrated to ensure they did appropriate analysis.

  3. Based on this approach it was possible that moving a Rule Family to a task may treat a Rule Family as a non-version or version change based on the selected Decision for which it was edited.

  4. Performing a non-version change on a Decision will result in the versioning of the Decision unless all Rule Families within the Decision have had their messages migrated already.

With Version 7.2.9 some enhancements have been made to simplify for users when a Rule Family will have its messages migrated and provide indications that this is occurring. The changes do require some data migrations on how logic changes are identified and marking a Rule Family as to whether it has had its messages migrated. Depending on the number of Decision Flows, Decisions, and Rule Families the data migrations may take longer than typical for a version upgrade.

With the enhancements in Version 7.2.9 the Decision to Rule Family Message migrations will support the following approach:

  1. New Rule Families will immediately own messages and they cannot be different across Decisions.

  2. Any Rule Families that were already versioned and had their messages migrated or created new in Version 7 will be marked with an indicator that their messages were migrated.

  3. Any Rule Family that is edited for the first time and has not had their messages migrated will have their messages migrated as follows:

  • When copied to the task, all message categories for all Decisions that are associated to that Rule Family version will be copied with the Rule Family, including those in other communities not available in the current community, so that a complete set of message categories will be provided with the Rule Family to support all associated Decisions. This will ensure the Rule Family will have all message categories and messages and a modeler can remove or edit messages as required for the new Rule Family version.

  • Where multiple versions of the same Decision View use the Rule Family version, the latest Decision View’s messages will be copied as the most recent messages are assumed the most correct.

  • After selecting the latest Decision View messages, where different Decision Views have different messages for the same message category, then for each row in the Rule Family the message category text for that row is set to the most recently created Decision version.

  • If a message category existed on an old Decision version and was removed, it will still be copied to the Rule Family if the old version is currently associated with the Rule Family version being edited. All message categories for all current Decision associations are copied so that the modeler may have all information to assess what to retain. In the case the old message categories were not required it will be easier to remove the message category than rebuild all the messages if they were not copied when they were required.

  1. On the Rule Family laundry line at the top of the screen there will be an indicator that this Rule Family has messages migrated in this task so that a modeler will know to review the messages for the new version of the Rule Family. In addition, the Rule Family will have a Task Status indicating message migration in the Rule Family Task Tab.

Note: The Rule Family compare can be used to easily compare the Draft Rule Family with the prior version in other Decisions.

  1. On approval the Rule Family messages will be migrated to the Rule Family and the Rule Family will be marked to indicate its messages have been migrated. Note this is for internal purposes, it is not visible on the user interface.

  2. Any Rule Families that are in a Draft status in a Task that have not had their messages migrated will be updated, during the upgrade process, with any missing Message Categories that exist on approved Decisions with the same algorithm for copying a Rule Family to a task described above. If the Rule Family already has the message category it will not have any messages updated as a user in the Task may have made changes to the messages in the task.

  3. Any Decision that is copied to the task for a non-version change will be approved as a non-version change.

  4. Any Rule Family that is copied to a task that has already had its messages migrated will be treated as an unchanged asset, non-version change, or version change depending on how the Rule Family is edited while in the task.
